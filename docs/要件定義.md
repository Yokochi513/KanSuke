# 要件定義書 — 家庭用スケジュールアプリ（KanSuke）

## 1. 概要

### 1.1 背景・目的
これまで家庭内のスケジュール共有に **TimeTree** を利用していたが、API サービスの終了を機に、家庭内専用のスケジュール共有アプリを自作する。
公式公開・ストア配信は行わず、**家庭内運用に限定**する。

### 1.2 スコープ
- 家族メンバー間でのスケジュール記録・共有
- 「誰の予定か」が一目で判別できる表示
- 「仮の予定」と「確定予定」の区別
- iPhone（iOS 16）でストレスなく閲覧・操作できること

### 1.3 スコープ外（当面対応しない）
- 一般公開・アプリストア配信
- 外部カレンダー（Google Calendar 等）連携
- 課金・広告・アカウント招待などの一般向け機能
- 予定追加/変更の共有通知（※将来拡張候補。本バージョンでは対象外）

---

## 2. 前提・制約

| 項目 | 内容 |
| --- | --- |
| フレームワーク | Flutter（iOS / Android 両対応） |
| 対象 OS | iOS 16 以降 / Android（直近メジャー数バージョン） |
| 利用範囲 | 家庭内クローズド運用（一般公開なし。家族メンバーのみアクセス可） |
| データ基盤 | **Firebase（マネージドクラウド）**。Firestore / Authentication / Cloud Functions / FCM を利用 |
| 同期 | **Cloud Firestore** によるリアルタイム同期・オフライン同期 |
| 想定利用者数 | 家族数名規模（少人数前提） |

> ※ 基盤は当初「自前サーバ（NAS / VPS / ラズパイ）」を想定していたが、IT に不慣れな家族でも運用しやすいよう、サーバ運用が不要な **Firebase** を採用する方針に変更した（[基本設計.md](基本設計.md) §0 参照）。データは Google クラウド上に保管される。

---

## 3. 利用者・ロール

| ロール | 説明 |
| --- | --- |
| 家族メンバー | 予定の閲覧・登録・編集・削除が可能な利用者。各自に色などの識別子を割り当てる |

※家庭内運用のため、厳密な権限分離（管理者/一般など）は当面設けず、全メンバーが同等の操作権限を持つ前提とする。

---

## 4. 機能要件

### FR-1. スケジュールの記録（CRUD）
- 予定の **登録 / 閲覧 / 編集 / 削除** ができる。
- 1件の予定が持つ属性：
  - タイトル
  - 日付（および任意で時刻 / 終日フラグ）
  - 所有者（誰の予定か）
  - 予定種別（**仮 / 確定**）
  - メモ（任意）

### FR-2. 所有者の視覚的識別
- 各予定が「誰の予定か」を **一目で判別**できる。
- メンバーごとに **色（またはアイコン/ラベル）** を割り当て、一覧・カレンダー上で識別表示する。

### FR-3. 予定種別の区別（仮 / 確定）
- 予定は **「仮の予定」** と **「確定予定」** の2種類を持つ。
- 種別を視覚的に区別する（例：仮は点線枠・半透明、確定は実線/塗りつぶし 等）。
- 仮 → 確定への状態変更が容易に行える。

### FR-4. カレンダー表示
- 月表示を基本とし、日付ごとに登録された予定を一覧できる。
- メンバー色・予定種別の区別が表示に反映される。
- （任意拡張）週/日表示の切り替え。

### FR-5. 予定前リマインド通知
- 予定の **開始前に、所有者（および希望者）へプッシュ通知**でリマインドする。
- リマインドのタイミング（例：当日朝 / 開始◯分前）を予定単位または既定設定で指定できる。
- ※「予定の追加・変更の共有通知」は本バージョン対象外。

### FR-6. データ同期
- **Cloud Firestore** を介して家族間でスケジュールデータを共有・同期する。
- 端末で行った変更が Firestore へ反映され、他端末へリアルタイムに伝播する。
- オフライン時の変更はローカルに保持され、オンライン復帰時に自動同期される。

## 5. 非機能要件

### NFR-1. パフォーマンス / ユーザビリティ
- **iPhone（iOS 16）でストレスなく閲覧・操作できる**こと（表示の引っかかり・過度な待ち時間がない）。
- カレンダーの月切り替え・予定一覧の表示が体感的に高速であること。

### NFR-2. 対応プラットフォーム
- iOS 16 以降および Android で同等の機能・操作性を提供する（Flutter による単一コードベース）。

### NFR-3. 可用性・データ保全
- Firestore のデータは **定期エクスポート**および **PITR（Point-in-Time Recovery）** により消失リスクを抑える。
- オフライン時・通信不達時でも、端末側でキャッシュ済みの予定は閲覧・編集でき、復帰時に自動同期される。

### NFR-4. セキュリティ
- 家庭内クローズド運用とし、通信は暗号化（Firebase SDK の TLS）する。
- 家族メンバーのみがアクセスできる認証を設ける（Firebase Authentication によるサインインと、家族メールの許可リストによるアクセス制限）。

### NFR-5. 保守性
- 個人/家庭での継続運用を前提に、構成をシンプルに保つ。
- サーバ・OS・DB の運用を不要にするため **Firebase（マネージドサービス）への依存を許容**する。保守対象は Cloud Functions と Security Rules に限定する。

---

## 6. データモデル（概要）

```
User（家族メンバー）
  - id
  - name
  - color（識別色）

Event（予定）
  - id
  - title
  - ownerId        → User.id
  - startAt / endAt（または date + allDay）
  - type           → { tentative（仮）, confirmed（確定）}
  - memo（任意）
  - reminder       → リマインド設定（任意）
  - createdAt / updatedAt
```

---

## 7. 想定システム構成（概要）

```
[ Flutter App (iOS / Android) ]
   │  サインイン         │  読み書き（リアルタイム / オフライン同期）
   ▼                     ▼
[ Firebase Authentication ]   [ Cloud Firestore ]   ← Security Rules で家族限定
   (Google / Apple サインイン)        │
                                       │ onWrite トリガ
                                       ▼
                              [ Cloud Functions ]   ← 定期実行でリマインド判定
                                       │
                                       ▼
                                    [ FCM ]   → 予定前リマインドを各端末へ配信
```

- 中間 API サーバは持たず、アプリは Firestore SDK で直接読み書きし、認可は Security Rules がサーバ側で担保する。
- プッシュ通知の配信経路は **FCM**（iOS は FCM 経由で APNs に中継）とする。
- 詳細は [基本設計.md](基本設計.md) を参照。

## 8. 確定事項 / 今後の検討課題

### 8.1 基本設計フェーズで確定した事項

| # | 項目 | 決定内容 |
| --- | --- | --- |
| 1 | サーバの稼働環境 | **Firebase（マネージド）** を採用。自前サーバ（NAS / VPS / ラズパイ）は不採用 |
| 2 | 認証方式 | **Firebase Authentication + Google / Apple サインイン**。家族メールの許可リストでアクセス制限 |
| 3 | リマインド配信経路 | **Cloud Functions（定期実行）→ FCM**（iOS は APNs に中継） |
| 4 | オフライン対応の範囲 | **閲覧・編集の両方**を許容。Firestore のオフライン同期で復帰時に自動反映 |

### 8.2 今後の検討課題

| # | 項目 | 補足 |
| --- | --- | --- |
| 5 | 繰り返し予定の要否 | 毎週・毎月などの繰り返しパターン対応（現バージョン対象外） |
| 6 | 将来拡張 | 予定追加/変更の共有通知、週/日表示、外部カレンダー連携 |
| 7 | コスト監視 | Cloud Functions 利用のため Blaze プラン（従量課金）。予算アラートで想定外課金を防止 |
