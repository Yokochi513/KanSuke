# ストア公開手順（Google Play / App Store）

> **現時点では一般公開・ストア配信は「スコープ外」**（[要件定義.md](../要件定義.md) 1.3）。
> 本書は「将来公開することになった場合に何が必要か」を先回りして整理した**参考資料**であり、
> 公開を決定した事実を意味しない。実際に公開する場合は、まず要件定義 1.2 / 1.3 の変更を Issue で議論すること（仕様書はエージェントが勝手に書き換えない — [AGENTS.md](../../AGENTS.md) 作業ルール 4）。
>
> 家庭内配布のみで済む場合は [Androidデプロイ手順.md](Androidデプロイ手順.md) / [iOSデプロイ手順.md](iOSデプロイ手順.md) を参照。

## 0. 最初に読むこと: 現状のままでは両ストアとも審査に落ちる

ビルドや証明書の話をする前に、**KanSuke の現在の設計にはストア審査を通過できない構造的な問題が 3 つある**。これらは配布手順ではなくアプリの仕様そのものの変更を要するため、公開を検討する時点で最初に扱う必要がある。

### ブロッカー 1: 審査員がサインインできない（最重要）

`functions/allowlist.js` の `evaluateSignup()` は、Firestore の `allowlist/{email}` に存在しないメールアドレスのサインアップをすべて拒否する。Auth Blocking Function として動くため、**家族以外は誰一人アカウントを作れない**。

これは家庭内運用としては正しい設計だが、ストア審査では致命的になる。

- **Apple**: App Review ガイドライン 2.1（App Completeness）— レビュー担当者がアプリの全機能を確認できる**デモアカウント**の提供が必須。提供しないとリジェクトされる。
- **Google Play**: 同様に、ログインが必要なアプリはレビュー用の認証情報の提供が必須。

対応の選択肢は次のいずれか。

| 選択肢 | 内容 | 備考 |
| --- | --- | --- |
| A. 審査用アカウントを allowlist に追加 | レビュー用のメールアドレスを `allowlist` に登録し、その資格情報を審査情報欄に記入 | 最小の変更。ただし**アプリが「招待された人しか使えない」こと自体**が Apple ガイドライン 4.2（Minimum Functionality）や 3.2 に抵触しうる。一般ユーザーがダウンロードしても何もできないアプリはストアに置く意味を問われる |
| B. 招待制モデルへ再設計 | 誰でもサインアップでき、カレンダーには招待経由でのみ参加できる形にする | ストアに出すなら本質的にはこれ。allowlist（家族固定）→ カレンダー単位の招待（[要件定義.md](../要件定義.md) 1.3 で「スコープ外」とされている**アカウント招待機能**そのもの）への作り替えが必要 |
| C. 公開しない | 現行の配布方法を続ける | |

**B は要件定義のスコープ外機能の実装を意味する。** つまり「ストア公開する」という判断は、実質的に「招待機能を作る」という判断とセットになる。ここを曖昧にしたまま手順だけ進めても審査で止まる。

### ブロッカー 2: アプリ内アカウント削除機能が無い

`lib/features/` 以下にアカウント削除の実装は見当たらない（`auth/` `settings/` ともに該当処理なし）。

- **Apple**: ガイドライン 5.1.1(v) — アカウント作成をサポートするアプリは、**アプリ内から**アカウント削除を開始できなければならない。設定画面への導線だけでなく、実際に削除が完了する経路が必要。
- **Google Play**: アカウント削除ポリシー — アプリ内の導線に加え、**ウェブ上からもアカウント削除を要求できる URL** の提出が必要。

実装としては Firebase Auth の `User.delete()` に加え、Firestore 上の当該ユーザーのデータ（`users/{uid}`、参加中カレンダーのメンバー情報、作成したイベント）の扱いを決める必要がある。共有カレンダーなので「本人を消すと他の家族の予定表示が壊れないか」の設計判断が要る。

### ブロッカー 3: プライバシーポリシーが存在しない

リポジトリ内にプライバシーポリシーに相当する文書は無い。両ストアとも**公開 URL の提出が必須**。

KanSuke は Firebase を使うため、最低限これらの収集を申告することになる。

| 収集するもの | 経路 | 用途 |
| --- | --- | --- |
| メールアドレス | Firebase Authentication（Google / Apple サインイン） | アカウント識別・allowlist 照合 |
| ユーザー ID | Firebase Authentication | アカウント識別 |
| デバイストークン（FCM 登録トークン） | `firebase_messaging`（Issue #13） | リマインド通知の配信（FR-5） |
| ユーザー生成コンテンツ（予定のタイトル・日時・メモ） | Cloud Firestore | アプリ機能の提供 |

GitHub Pages 等で静的ページとして公開すれば足りる。

## 1. 公開する場合の共通準備

ブロッカー 1〜3 を解消したうえで、両ストア共通で必要になるもの。

### 1-1. アプリ識別子とブランド

- **Application ID / Bundle ID**: 現在どちらも `com.kansuke.kansuke`（`android/app/build.gradle.kts` および `ios/Runner.xcodeproj`）。
  - **一度ストアに公開すると変更できない。** `com.kansuke.*` は所有ドメインに基づかない ID なので、公開前に自分の管理下にあるドメイン由来の ID（例 `dev.<自分のドメイン>.kansuke`）へ変更すべきか検討する。変更する場合は Firebase 側のアプリ登録もやり直し、`flutterfire configure` で `lib/firebase_options.dart` を再生成する。
- **アプリ名**: 「KanSuke」が既存の商標・既存アプリと衝突しないか確認する。App Store は名前が全世界で一意。

### 1-2. ストア掲載素材

- アイコン: `flutter_launcher_icons` で生成済み（`assets/icon/app_icon.png`）。ストア用には別途 1024×1024 の PNG（**アルファチャンネルなし**、App Store の要件）が要る。
- スクリーンショット: iOS は 6.7 インチと 6.5 インチ（もしくは現行の必須サイズ）、Android は最低 2 枚。
- 説明文、短い説明、プロモーションテキスト、サポート URL。
- 年齢レーティング設問への回答。

### 1-3. バージョニング

`pubspec.yaml` の `version:`（現在 `1.1.0+2`）が両プラットフォームの版数の単一の出所になる。

- `1.1.0` → Android の `versionName` / iOS の `CFBundleShortVersionString`
- `+2` → Android の `versionCode` / iOS の `CFBundleVersion`

**ビルド番号は両ストアとも単調増加が必須**で、一度使った番号は再利用できない。

### 1-4. FR-7（自作アップデート通知）の見直し

現在 `lib/features/version_check/` は Firestore の `meta/release` を見て「新しいバージョンがあります」と促す（[要件定義.md](../要件定義.md) FR-7）。これは APK 手動配布のために作られた仕組み。

ストア公開後はストアの自動更新が効くため不要になるうえ、**Apple はアプリ内からストア外の更新へ誘導する挙動を嫌う**（ガイドライン 2.5.2 / 3.1.1）。`url_launcher` で外部に飛ばす作りになっている場合は、公開前にストアのアプリページへ向けるか、機能自体を落とすかを決める。

## 2. Google Play への公開

### 2-1. デベロッパー登録

<https://play.google.com/console/> で登録。**登録料 $25（買い切り、初回のみ）**。

> **注意（個人アカウントの場合）**: 2023 年以降に作成された個人デベロッパーアカウントは、本番リリースの前に**一定人数のテスターによるクローズドテストを一定期間継続する**ことが求められる（人数・日数の要件は変更されることがあるため、必ず Play Console の最新の案内を確認すること）。家族数名しかいない状況ではこの人数要件を満たせない可能性があり、**公開の実現性を左右する**。組織アカウントにはこの要件は適用されない。

### 2-2. アプリ署名

Play では **Play App Signing** を使う（アップロード鍵で署名 → Google が配信用鍵で再署名）。

既に `kansuke-release-key.jks` を作成済み（[Androidデプロイ手順.md](Androidデプロイ手順.md) 手順 1）なので、これを**アップロード鍵**として登録できる。

> リポジトリ直下に `kansuke-release-key.jks` が置かれているが、`.gitignore` の `*.jks` により追跡対象外であり、コミット履歴にも含まれていないことを確認済み（`git log --all -- '*.jks'` が空）。この状態を維持すること。

### 2-3. ビルド

ストア配信は APK ではなく **AAB（Android App Bundle）**。

```bash
flutter build appbundle --release
```

出力: `build/app/outputs/bundle/release/app-release.aab`

- Play Console の要件で **targetSdk は一定以上**が求められる（毎年引き上げられる）。現在は `flutter.targetSdkVersion` に委ねているので、Flutter SDK の更新で追随する形になる。

### 2-4. Play Console での設定

1. アプリを作成（名前、デフォルト言語、アプリ or ゲーム、有料 or 無料）。
2. **データセーフティ（Data safety）フォーム**に回答 — ブロッカー 3 の表の内容をそのまま申告する。虚偽申告は公開停止の対象。
3. プライバシーポリシー URL を登録。
4. **アカウント削除 URL** を登録（ブロッカー 2）。
5. アプリのコンテンツ（広告の有無、対象年齢、コンテンツレーティング）に回答。
6. レビュー用のログイン情報を「アプリのアクセス権」に記入（ブロッカー 1）。

### 2-5. リリーストラック

`内部テスト` → `クローズドテスト` → `オープンテスト` → `製品版` の順に上げていく。内部テストは審査なしで即配信でき、家族のみへの配布に使える。

> **家庭内運用のままで良いなら**、Play Console の**内部テストトラックに留める**という選択肢もある。$25 で最大 100 名に配信でき、審査もデータセーフティ以外は緩い。ブロッカー 1〜3 の多くを回避しつつ、APK 手渡しより楽な配布ができる。**現実的な折衷案としてはこれが最も有力。**

## 3. Apple App Store への公開

### 3-1. Apple Developer Program

**年額 $99（毎年更新）**。[iOSデプロイ手順.md](iOSデプロイ手順.md) の 1-1 と同じで、TestFlight 配布時点で既に加入済みのはず。

### 3-2. ビルドとアップロード

TestFlight 配布と全く同じ。

```bash
flutter build ipa --release
```

`build/ios/archive/Runner.xcarchive` を Xcode Organizer から **Distribute App → App Store Connect** でアップロードする。

### 3-3. 暗号化に関する申告

HTTPS 通信しかしていない場合、毎回の申告を省くため `ios/Runner/Info.plist` に追記しておくとよい。

```xml
<key>ITSAppUsesNonExemptEncryption</key>
<false/>
```

> これは「標準的な暗号化しか使っていない」という申告。独自の暗号実装を追加した場合は再検討が必要。

### 3-4. App Store Connect での設定

1. **App Privacy（プライバシー栄養ラベル）** — ブロッカー 3 の表の内容を申告。「データはユーザーに紐づく」形になる点に注意（メールアドレス・ユーザー ID と結びつくため）。
2. プライバシーポリシー URL。
3. **App Review Information** にデモアカウントのメールアドレスとパスワードを記入（ブロッカー 1）。
   - Google / Apple サインインしか実装していないため、**審査員が使える資格情報を渡せない**という問題が起きる。メール+パスワードのサインイン経路を審査用に用意するか、あるいはサインイン不要で試せるデモモードを設ける必要がある。ここは実装作業を伴う。
4. 「Notes」欄に、アプリが家族向けの共有カレンダーであること、デモアカウントでの操作手順を書いておくと審査が通りやすい。

### 3-5. 審査

TestFlight にビルドを上げた後、「App Store 提出用」として審査に出す。通常は数日以内。リジェクトされた場合は Resolution Center で理由が示されるので、修正して再提出する。

**KanSuke で特にリジェクト理由になりやすい項目**:

| ガイドライン | 内容 | KanSuke での該当 |
| --- | --- | --- |
| 2.1 App Completeness | デモアカウント未提供 | ブロッカー 1 |
| 4.2 Minimum Functionality | 特定の人しか使えない / 機能が乏しい | allowlist モデルそのもの |
| 5.1.1(v) Account Deletion | アプリ内でアカウント削除できない | ブロッカー 2 |
| 4.8 Login Services | 第三者サインイン提供時は Sign in with Apple も必須 | **対応済み**（`sign_in_with_apple` 導入済み・`Runner.entitlements` に entitlement あり） |
| 5.1.1 Data Collection | プライバシーポリシー未提供 | ブロッカー 3 |

## 4. コスト・運用負荷のまとめ

| | Google Play | App Store |
| --- | --- | --- |
| 費用 | $25（買い切り） | $99 / 年（更新しないと配信停止） |
| 審査 | あり（内部テストトラックは実質なし） | あり（TestFlight 内部テスターは審査なし） |
| 個人アカウントの追加要件 | クローズドテストの人数・期間要件あり | なし |
| 公開後の負荷 | targetSdk 引き上げへの毎年の追随 | Xcode / SDK 更新への追随、年額更新 |

加えて **Firebase の Blaze プランは従量課金**である。家族数名なら無料枠に収まるが、一般公開してユーザーが増えれば Firestore の読み取り課金が直接効いてくる。公開する場合は予算アラートの設定と、Firestore セキュリティルール（現在は家族クローズド前提）の全面的な見直しが必須になる。

## 5. 推奨する進め方

1. **まず「本当に一般公開が必要か」を Issue で問う。** 家族以外に配る必要がないなら、Play の内部テストトラック（$25）+ TestFlight（$99/年）に留めるのが、審査もプライバシー申告も最小で済む。
2. 一般公開する場合は、**配布手順ではなく仕様変更（ブロッカー 1〜3）から着手する**。allowlist → 招待制への転換は要件定義 1.3 のスコープ変更にあたるため、実装前に Issue で合意を取る。
3. 仕様が固まったら、プライバシーポリシーとアカウント削除を先に実装し、その後でストア登録作業に進む。
