# リリース手順（develop → main）

> `develop` に溜まった変更を `main` に出し、家族の端末へ配るまでの運用手順。
> 準備作業（バージョン更新・CHANGELOG 記載・APK ビルド・リリース PR 作成）は
> Claude Code の skill `release-kansuke`（`.claude/skills/release-kansuke/SKILL.md`）で自動化している。
> **本書は skill が担当しない「人間がやること」を含む全体像**を示す。
> 端末へのインストール方法は [Androidデプロイ手順.md](Androidデプロイ手順.md) / [iOSデプロイ手順.md](iOSデプロイ手順.md) を参照。
> **バックエンド（Functions / Rules / インデックス）のデプロイは本書の対象外**で、
> リリースとは別作業として [Firebaseデプロイ手順.md](Firebaseデプロイ手順.md) を実行する。

## 全体の流れ

```
1. リリース準備            … skill: release/x.y.z ブランチ + PR 作成
2. リリース PR をマージ    … 人間
3. CI の反映を確認         … 人間
4. APK を家族に配布        … 人間
5. main → develop back-merge … 人間（必須。忘れるとバージョンがずれる）
```

## 1. リリース準備（skill が実行）

- `pubspec.yaml` のバージョンを上げる（例 `1.2.0+3` → `1.3.0+4`。ビルド番号は毎回 +1）
- `CHANGELOG.md` の `## [Unreleased]` を `## [x.y.z] - YYYY-MM-DD` にリネームし、新しい空の `## [Unreleased]` を先頭に追加（FR-7）
- `flutter analyze` / `flutter test` / `dart format` チェック
- `flutter build apk --release` でリリース署名済み APK をビルド
- `release/x.y.z` ブランチを `main` 向け PR にする（タイトル `release: vX.Y.Z`）

## 2. リリース PR をマージ（人間）

GitHub 上で PR をレビューし、**マージコミットを作る形でマージ**する（`Merge pull request #NN from Yokochi513/release/x.y.z`）。
squash すると後述の back-merge で差分が二重に見えるため、squash は使わない。

## 3. CI の反映を確認（人間）

`main` への push で 3 つのワークフローが走る。

| ワークフロー | 内容 | 確認方法 |
| --- | --- | --- |
| `publish-release-version.yml` | CHANGELOG の該当バージョンを読み取り、Firestore の `meta/release`（最新1件・起動時のお知らせ用）と `releases/{version}`（更新履歴・全バージョン、Issue #96）の両方に反映 | Actions が成功し、`meta/release` の `version` が新バージョンになっていること。アプリを開くとバージョンアップのお知らせが出る。設定 →「更新履歴」に新バージョンが最上段で並ぶ |
| `deploy-pages.yml` | Web 版を GitHub Pages にデプロイ | Actions が成功し、公開ページが更新されていること |
| `upload-play-internal.yml` | 署名済み AAB をビルドし、Play Console の**内部テストトラック**へアップロード。`pubspec.yaml` が変わった push でだけ走る | Actions が成功し、Play Console の内部テスト → リリース一覧に新バージョンが出ていること。セットアップは本書末尾の「付録: Play Console 自動アップロードのセットアップ」を参照 |

## 4. アプリを家族に配布（人間）

Play Console の内部テストトラックを使う場合、AAB は `upload-play-internal.yml` が自動でアップロードするので、家族は Play ストアから更新するだけでよい。内部テストの参加リンクを一度共有しておけば以降の配布作業は不要。

Play を使わず APK を手渡しする場合は、skill がビルドした `build/app/outputs/flutter-apk/app-release.apk` を家族の Android 端末に渡してインストールする（手順は [Androidデプロイ手順.md](Androidデプロイ手順.md)）。

- APK・AAB・`kansuke-release-key.jks`・`android/key.properties` は**コミットしない**（`.gitignore` 済み）。
- リリース署名鍵で署名されているため、既存アプリに上書きインストールできる。デバッグ署名の APK を配ると署名不一致でインストールに失敗するので注意。

## 5. main → develop の back-merge（人間・必須）

リリース準備コミット（バージョン更新・CHANGELOG）は `release/x.y.z` 上にあり、`main` にしか入っていない。
**そのままだと `develop` のバージョンが古いまま**になり、次のリリースで差分がおかしくなる。マージ後すみやかに `main` を `develop` に戻す。

```bash
git checkout develop
git pull                      # origin/develop を最新化
git fetch origin main
git merge origin/main         # コンフリクトは通常起きない
git push origin develop
```

コミットメッセージは実績に合わせて `Merge branch 'main' into develop（vX.Y.Z リリース back-merge）` とする（例: `09fc0cf`）。

back-merge 後、`develop` で以下が `main` と一致していることを確認する。

- `pubspec.yaml` の `version:` が新バージョン
- `CHANGELOG.md` の先頭が空の `## [Unreleased]`、その下が `## [x.y.z]`

## チェックリスト

- [ ] リリース PR をマージした（squash ではなくマージコミット）
- [ ] `publish-release-version.yml` が成功し、`meta/release` が新バージョンになった（`releases/{version}` にも履歴が追加された）
- [ ] `deploy-pages.yml` が成功した
- [ ] `upload-play-internal.yml` が成功し、Play の内部テストに新バージョンが出た（Play を使う場合）
- [ ] アプリを家族に配布した（Play 内部テスト、または APK 手渡し）
- [ ] `main` を `develop` に back-merge して push した

## 付録: Play Console 自動アップロードのセットアップ（人間・一度だけ）

`upload-play-internal.yml` が動くには、以下を**先に人間が済ませておく必要がある**。Firebase 同様、Console 上の操作はエージェントに代行させない。

### A. Play Console 側

1. [Play Console](https://play.google.com/console/) にデベロッパー登録（**$25 買い切り**）し、アプリを作成する。パッケージ名は `com.kansuke.kansuke`（`android/app/build.gradle.kts` の `applicationId` と一致させる。一度公開すると変更できない）。
2. **AAB を 1 本だけ手動でアップロードする。** API 経由の初回アップロードは受け付けられないため、最初は Console からアップロードして内部テストのリリースを 1 つ作る。以降は CI に任せられる。
   - ローカルでのビルド: `flutter build appbundle --release` → `build/app/outputs/bundle/release/app-release.aab`
3. アップロード鍵として `kansuke-release-key.jks` が登録される（Play App Signing。配信用鍵は Google が保持する）。**この鍵を失うと以降アップロードできなくなる**ので、バックアップを取る。
4. 内部テストのテスターに家族のメールアドレスを登録し、参加リンクを共有する。

### B. サービスアカウント（Play Developer API）

> かつて存在した Play Console の「設定 → API アクセス」ページは**廃止済み**。デベロッパーアカウントと Google Cloud プロジェクトを紐づける操作も不要になった（[公式](https://developers.google.com/android-publisher/getting_started)）。作業のほとんどは Google Cloud Console 側で行う。

1. [Google Cloud Console](https://console.cloud.google.com/) で **サービスアカウントを作成**する（`IAM と管理 → サービスアカウント → サービスアカウントを作成`）。プロジェクトは任意。既存の Firebase プロジェクト `kansuke-b6d32` を使ってよい。
2. 同じプロジェクトで **Google Play Android Developer API** を有効化する（`API とサービス → ライブラリ` で検索）。
3. 作成したサービスアカウントの **JSON 鍵**をダウンロードする（`キー → 鍵を追加 → 新しい鍵を作成 → JSON`）。
4. **Play Console → ユーザーと権限 → 新しいユーザーを招待**で、サービスアカウントのメールアドレス（`...@....iam.gserviceaccount.com`）を入力し、KanSuke に対して**「リリースを作成」「製品版以外のトラックへのリリース」**の権限を与える。権限が反映されるまで数分〜数時間かかることがある。

### C. GitHub Secrets（リポジトリ設定 → Secrets and variables → Actions）

署名鍵と Firebase 設定はリポジトリに含めていない（`.gitignore` 済み）ため、CI では Secrets から復元する。

| Secret 名 | 中身 | 作り方 |
| --- | --- | --- |
| `PLAY_SERVICE_ACCOUNT_JSON` | B でダウンロードした JSON の中身そのまま | ファイルを開いて全文貼り付け |
| `ANDROID_KEYSTORE_BASE64` | `kansuke-release-key.jks` を base64 化した文字列 | Git Bash: `base64 -w0 kansuke-release-key.jks \| gh secret set ANDROID_KEYSTORE_BASE64`（PowerShell の `[IO.File]::ReadAllBytes` は相対パスを PowerShell のカレントで解決しないため、使うなら絶対パスを渡すこと） |
| `ANDROID_KEYSTORE_PASSWORD` | `android/key.properties` の `storePassword` | そのまま |
| `ANDROID_KEY_PASSWORD` | 同 `keyPassword` | そのまま |
| `ANDROID_KEY_ALIAS` | 同 `keyAlias`（`kansuke`） | そのまま |
| `ANDROID_GOOGLE_SERVICES_JSON` | `android/app/google-services.json` の中身そのまま | ファイルを開いて全文貼り付け |

### 注意

- **`versionCode`（`pubspec.yaml` の `+N`）は再利用できない。** 同じ番号の AAB を再アップロードすると Play が 409 で拒否する。ワークフローを手動で再実行しても同じ理由で失敗するので、失敗時はバージョンを上げてから出し直す。
- ワークフローは `pubspec.yaml` が変わった `main` への push でだけ走る。バージョンを上げないコミットを `main` に直接入れてもアップロードは起きない。
