# リリース手順（develop → main）

> `develop` に溜まった変更を `main` に出し、家族の端末へ配るまでの運用手順。
> 準備作業（バージョン更新・CHANGELOG 記載・APK ビルド・リリース PR 作成）は
> Claude Code の skill `release-kansuke`（`.claude/skills/release-kansuke/SKILL.md`）で自動化している。
> **本書は skill が担当しない「人間がやること」を含む全体像**を示す。
> 端末へのインストール方法は [Androidデプロイ手順.md](Androidデプロイ手順.md) / [iOSデプロイ手順.md](iOSデプロイ手順.md) を参照。

## 全体の流れ

```
1. リリース準備            … skill: release/x.y.z ブランチ + PR 作成
2. リリース PR をマージ    … 人間
3. CI の反映を確認         … 人間
4. APK を家族に配布        … 人間
5. main → develop back-merge … 人間（必須。忘れるとバージョンがずれる）
```

## 1. リリース準備（skill が実行）

- `pubspec.yaml` のバージョンを上げる（例 `1.2.0+3` → `1.3.0+4`。ビルド番号は毎回 +1）
- `CHANGELOG.md` の `## [Unreleased]` を `## [x.y.z] - YYYY-MM-DD` にリネームし、新しい空の `## [Unreleased]` を先頭に追加（FR-7）
- `flutter analyze` / `flutter test` / `dart format` チェック
- `flutter build apk --release` でリリース署名済み APK をビルド
- `release/x.y.z` ブランチを `main` 向け PR にする（タイトル `release: vX.Y.Z`）

## 2. リリース PR をマージ（人間）

GitHub 上で PR をレビューし、**マージコミットを作る形でマージ**する（`Merge pull request #NN from Yokochi513/release/x.y.z`）。
squash すると後述の back-merge で差分が二重に見えるため、squash は使わない。

## 3. CI の反映を確認（人間）

`main` への push で 2 つのワークフローが走る。

| ワークフロー | 内容 | 確認方法 |
| --- | --- | --- |
| `publish-release-version.yml` | CHANGELOG の該当バージョンを読み取り Firestore `meta/release` に反映 | Actions が成功し、`meta/release` の `version` が新バージョンになっていること。アプリを開くとバージョンアップのお知らせが出る |
| `deploy-pages.yml` | Web 版を GitHub Pages にデプロイ | Actions が成功し、公開ページが更新されていること |

## 4. APK を家族に配布（人間）

skill がビルドした `build/app/outputs/flutter-apk/app-release.apk` を家族の Android 端末に渡してインストールする（手順は [Androidデプロイ手順.md](Androidデプロイ手順.md)）。

- APK・`kansuke-release-key.jks`・`android/key.properties` は**コミットしない**（`.gitignore` 済み）。
- リリース署名鍵で署名されているため、既存アプリに上書きインストールできる。デバッグ署名の APK を配ると署名不一致でインストールに失敗するので注意。

## 5. main → develop の back-merge（人間・必須）

リリース準備コミット（バージョン更新・CHANGELOG）は `release/x.y.z` 上にあり、`main` にしか入っていない。
**そのままだと `develop` のバージョンが古いまま**になり、次のリリースで差分がおかしくなる。マージ後すみやかに `main` を `develop` に戻す。

```bash
git checkout develop
git pull                      # origin/develop を最新化
git fetch origin main
git merge origin/main         # コンフリクトは通常起きない
git push origin develop
```

コミットメッセージは実績に合わせて `Merge branch 'main' into develop（vX.Y.Z リリース back-merge）` とする（例: `09fc0cf`）。

back-merge 後、`develop` で以下が `main` と一致していることを確認する。

- `pubspec.yaml` の `version:` が新バージョン
- `CHANGELOG.md` の先頭が空の `## [Unreleased]`、その下が `## [x.y.z]`

## チェックリスト

- [ ] リリース PR をマージした（squash ではなくマージコミット）
- [ ] `publish-release-version.yml` が成功し、`meta/release` が新バージョンになった
- [ ] `deploy-pages.yml` が成功した
- [ ] APK を家族に配布した
- [ ] `main` を `develop` に back-merge して push した
